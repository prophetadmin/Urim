# Knowledge Cockpit

Project concept: Knowledge Cockpit

Core idea

Build a local-first retrieval application for conversational knowledge-base querying. The application should let a user ask natural-language questions, retrieve grounded material from the local knowledge base, and receive a natural answer generated by a local reasoning model. The experience should feel polished and conversational rather than like a raw search console.

Primary interaction model

- Chat-first interface.
- User asks questions in normal language.
- Retrieval happens behind the scenes.
- Answers should read naturally when support is strong.
- If support is weak or insufficient, the system must say so plainly.
- Sources are returned explicitly in chat only when the user asks for them.

Primary product intent

- Informal research over a local knowledge base.
- Grounded answers rather than free hallucinated chat.
- Local inference and local retrieval as the default operating model.
- User-triggered session saving rather than automatic logging of every chat.

Reasoning model boundary

- The application should use the currently exposed local llama.cpp chat API.
- Model switching is outside the application and handled by existing local tray/service logic.
- The application should not implement internal model switching.

Retrieval architecture

The retrieval system is not a single-step text search. It is a two-stage pointer-based retrieval design.

Confirmed chain:

1. Query text is embedded locally through the dedicated llama.cpp embedding service.
2. The embedding vector is sent to a Qdrant collection.
3. Qdrant returns nearest-neighbor hits with metadata such as `tier`, `source_id`, `parent_id`, and `chunk_index`.
4. Qdrant does not return readable authority text for the tested collection.
5. Readable authority text is resolved separately from boxed markdown artifacts on disk.
6. The reasoning model answers from resolved authority text, not from Qdrant payload alone.

Current verified runtime shape

- Chat service:
  - local llama.cpp server on `127.0.0.1:8081`
- Embed service:
  - local llama.cpp embedding server on `127.0.0.1:8084`
  - verified endpoint: `POST /embedding`
- Vector database:
  - local Qdrant on `127.0.0.1:6333`
- Verified collection:
  - `emb_v2_nomic_embed_text_v1_5_f16_768`

Lexicon role

Lexicon should act as an advisory grounding layer before retrieval. It should help narrow or clarify candidate meaning for exact terms, aliases, redirects, or entity-like matches. It should not replace semantic retrieval and it should not hard-block retrieval when it contributes nothing useful.

Best current mental model:

- user query
- Lexicon assist
- local embedding
- Qdrant nearest-neighbor search
- boxed authority resolution
- support evaluation
- local reasoning answer

Authority text resolution

The readable knowledge source is stored separately from Qdrant in boxed artifacts on disk.

Verified boxed pattern for tested non-Wikipedia tiers:

- `D:\boxed\TierL_EB\<source_id>\<source_id>.md`
- `D:\boxed\Tier2\<source_id>\<source_id>.md`

Associated files also exist alongside authority markdown, including:

- `<source_id>.meta.json`
- `<source_id>.sha256`

This means the application needs a boxed resolver subsystem that maps retrieval hit metadata to authoritative boxed markdown by `tier` and `source_id`.

Important architectural principle

- Qdrant is a vector locator, not the authority text store.
- Boxed markdown is the readable authority source.
- Stable SHA-based identity is the key retrieval anchor.
- Physical storage roots may move, but retrieval remains valid if identity continuity and resolver mapping are preserved.

Support policy

The application should not bluff.

- If retrieval support is strong, answer naturally.
- If retrieval support is partial or weak, answer with an explicit caveat.
- If retrieval support is insufficient, say that sufficient support was not found.
- The system should not present unsupported claims as grounded KB answers.

Source disclosure policy

- Do not clutter normal answers with automatic source dumps.
- If the user asks for sources, return them explicitly in the chat window.
- Source disclosure should come from resolved boxed authority content and associated metadata.

Session policy

- Session saving should be user-triggered.
- Some sessions may be useless and should not be persisted automatically.
- Saved sessions may later contribute to higher-level summaries, but summary generation is not part of the first live interaction loop.

Chunk handling direction

Chunk-oriented reasoning is the preferred design target because local-model reasoning should operate over retrieved support units rather than blindly over whole documents. However, full document resolution is already proven via boxed markdown, so chunk slicing or excerpt extraction can be built on top of that authority layer.

POC shape

The first proof of concept should prioritize one clean workflow:

1. user asks a question
2. system retrieves grounded material
3. system answers naturally if support is sufficient
4. user can request sources in chat
5. user can optionally save the session

The POC does not need to start as a heavy analyst console or a desktop-native application. A locally hosted web application remains the strongest first shape because it is easier to iterate and easiest to integrate with the already-running local services.

What this project is trying to prove

- A local-first retrieval chat app can provide useful grounded research interaction over the existing KB.
- The current local infrastructure is sufficient to support end-to-end retrieval and reasoning.
- Pointer-based vector retrieval plus boxed authority resolution is a viable application architecture.
- Stable identity-based retrieval can survive storage movement as long as boxed content remains immutable and resolver mappings remain deterministic.

What this project is not trying to prove in the first pass

- Internal model orchestration or model switching UI
- A full analyst/debug lab
- Automatic source display in every answer
- Wikipedia boxed storage finalization
- Long-term optimal physical storage layout for tens of millions of boxed artifacts
- Production-scale storage repacking strategy

Current strongest seed-driving facts

- The app is chat-first.
- Reasoning is local.
- Embeddings are local.
- Retrieval is local.
- Qdrant hit payloads are pointer metadata, not readable answer text.
- Boxed authority files are the readable source of truth for tested non-Wikipedia tiers.
- Stable SHA-derived identities are fundamental to retrieval continuity.
- The app must include a resolver layer between Qdrant hits and answer generation.
- Weak evidence must be surfaced honestly.
- Source disclosure is on request, in chat.
